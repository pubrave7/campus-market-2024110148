二、发布表单字段设计

2.1 四类业务字段对比

我的设计遵循"贴合校园业务场景、字段必要且不冗余"的原则，四种发布类型的字段设计如下：

| 发布类型 | 数据集合 | 必填字段 | 选填字段 | 设计理由 |
|---------|---------|---------|---------|---------|
| 二手交易 | `trades` | 商品名称、价格、商品描述、发布人 | 分类、成色、所在校区、图片 | 价格是二手交易核心决策因素；成色影响购买意愿；分类便于筛选 |
| 跑腿委托 | `errands` | 标题、详细描述、酬劳、发布人 | 委托类型、截止时间、取件地点、送达地点、图片 | 酬劳直接影响接单意愿；取件/送达地点是跑腿任务的关键信息 |
| 拼单搭子 | `groupBuys` | 标题、详细描述、目标人数、发起人 | 拼单类型、截止时间、地点、图片 | 目标人数决定拼单能否成行，是核心字段；截止时间制造紧迫感 |
| 失物招领 | `lostFounds` | 标题、详细描述、联系方式 | 信息类型（寻物/招领）、物品名称、地点、日期、图片 | 联系方式是失物返还的唯一渠道，必须填写；信息类型用于区分寻物和招领 |

2.2 设计思路说明

1. **必填字段最小化原则**：只将业务核心字段设为必填（如价格、酬劳、联系方式），降低用户发布门槛；
2. **选填字段有默认值**：如地点默认为"未指定"、日期默认为当天，确保数据完整性；
3. **发布人使用固定占位**：当前阶段发布人字段由用户手动填写昵称，后续 Day5 可结合 Pinia 模拟登录态自动填充；
4. **图片上传为通用功能**：四种类型均可上传图片（base64 编码），统一放置于表单顶部，限制 2MB 大小；
5. **类型切换自动重置**：切换发布类型时清空所有已填写内容和校验错误，避免不同类型之间字段数据残留。

三、表单校验规则设计

3.1 校验策略

Day4 采用"字段级即时校验 + 提交前最终校验"的双重保障策略：

| 校验层次 | 触发时机 | 作用 |
|---------|---------|------|
| 字段级校验 | 提交时执行 `validateXxxForm()` | 逐字段检查，生成具体错误提示，显示在对应字段下方 |
| 提交按钮状态 | 通过 `checkFormValid()` 实时计算 | 必填字段全部填写后按钮才可点击（视觉反馈） |
| 最终校验 | `handleSubmit()` 开始时再次调用校验 | 防止绕过按钮状态直接触发表单提交 |

3.2 各类型校验规则

**二手交易（trade）**：
- 商品名称：非空（错误提示："请输入商品名称"）
- 价格：非空且数值大于 0（错误提示："请输入有效的价格（大于 0）"）
- 商品描述：非空（错误提示："请输入商品描述"）
- 发布人：非空（错误提示："请输入发布人昵称"）

**跑腿委托（errand）**：
- 标题：非空（错误提示："请输入委托标题"）
- 详细描述：非空（错误提示："请输入详细描述"）
- 酬劳：非空且数值大于 0（错误提示："请输入有效的酬劳金额"）
- 发布人：非空（错误提示："请输入发布人昵称"）

**拼单搭子（groupBuy）**：
- 标题：非空（错误提示："请输入拼单标题"）
- 详细描述：非空（错误提示："请输入详细描述"）
- 目标人数：非空且 ≥ 2（错误提示："目标人数至少为 2 人"）
- 发起人：非空（错误提示："请输入发起人昵称"）

**失物招领（lostFound）**：
- 标题：非空（错误提示："请输入标题"）
- 详细描述：非空（错误提示："请输入详细描述"）
- 联系方式：非空（错误提示："请输入联系方式"）

3.3 校验设计理由

- **数值字段增加"大于 0"判断**：价格和酬劳为 0 没有实际意义，加入最小值和大于 0 的判断比单纯检查非空更有意义；
- **目标人数 ≥ 2**：拼单至少需要 2 人才能成行，1 人拼单不符合业务逻辑；
- **不引入第三方校验库**：Day4 阶段保持校验逻辑简洁、可读，学生可以直观理解每一行校验代码；
- **错误信息具体化**：不说"请输入"，而是说"请输入商品名称"、"请输入有效的价格（大于 0）"，帮助用户快速定位问题字段。

四、数据新增实现

4.1 POST 请求封装

在 Day3 已有 API 模块基础上，为每类业务补充了 `create*()` 方法：

| API 文件 | POST 方法 | 请求路径 |
|---------|----------|---------|
| `src/api/trade.ts` | `createTrade(data)` | `POST /trades` |
| `src/api/errand.ts` | `createErrand(data)` | `POST /errands` |
| `src/api/groupBuy.ts` | `createGroupBuy(data)` | `POST /groupBuys` |
| `src/api/lostFound.ts` | `createLostFound(data)` | `POST /lostFounds` |

所有 POST 请求均使用 TypeScript 的 `Omit<T, 'id'>` 类型约束，不传 `id` 字段，由 JSON Server 自动生成自增 ID。

4.2 数据提交流程

```
用户选择发布类型 → 填写表单 → 点击"立即发布"
  → validateXxxForm() 逐字段校验
    → 校验失败：显示红色错误提示，阻止提交
    → 校验通过：
      → submitting = true（按钮显示"发布中..."，禁用点击）
      → 调用 createXxx(data) 发送 POST 请求
        → 成功（201）：
          → 显示 ✅ 成功提示
          → 重置表单
          → 1.5 秒后自动跳转到对应列表页
        → 失败：
          → 显示 ❌ 错误提示（提示检查 json-server 是否启动）
      → submitting = false
```

4.3 数据一致性保障

- 提交前对所有字符串字段执行 `.trim()` 去空格；
- 选填字段为空时填充默认值（如"未指定"）；
- 发布时间使用 `new Date().toISOString().slice(0, 10)` 自动生成；
- 拼单的 `currentCount` 固定初始化为 1（发起人自身）；
- 所有新增数据的 `status` 初始化为 `'open'`。

五、FormField 组件设计

创建了 `src/components/FormField.vue` 作为可复用的表单项包装组件：

**Props**：
- `label`：表单项标题
- `fieldId`：关联的 input id（点击标题可聚焦到输入框）
- `required`：是否必填（显示红色 * 标记）
- `error`：校验错误信息（非空时显示红色错误文本，同时为输入框添加红色边框）

**设计理由**：
- 统一四种发布类型的表单项展示风格；
- 避免在每个表单字段中重复编写 label、required 标记和 error 显示逻辑；
- 使用插槽（slot）放置实际的 input/textarea/select 控件，保持灵活性；
- 使用 `:deep()` 选择器为错误状态下的输入框添加红色边框样式。

6.2 人工审查与调整

| 审查项目 | AI 生成内容的问题 | 我的调整 |
|---------|-----------------|---------|
| 字段是否贴合业务 | AI 最初为二手交易增加了"品牌""型号"等电商字段 | 删除品牌/型号字段，校园二手交易场景下成色比品牌更重要 |
| 校验是否必要 | AI 为所有字段都加了校验（包括选填字段） | 只保留必填字段的校验，选填字段允许为空 |
| 提交路径是否正确 | AI 将 lostFound 的 POST 路径写成了 `/lost-found`（带连字符） | 修正为 `/lostFounds`，与 db.json 中的集合名一致 |
| 数据字段是否一致 | AI 生成的 errand 提交数据使用了 `location` 单一字段 | 修正为 `pickupLocation` 和 `deliveryLocation` 双字段，与 db.json 和列表页保持一致 |
| 跳转是否合理 | AI 建议提交后跳转到首页 | 改为跳转到对应类型的列表页（如发布二手交易后跳转 /trade），用户可立即看到新增数据 |
| 切换类型是否重置 | AI 未处理类型切换时的表单残留问题 | 添加了 `watch(activeType, ...)` 监听，切换类型时自动重置表单和校验错误 |

6.3 AI 使用体会

AI 在生成表单代码方面效率很高，能快速产出四套表单模板和校验函数框架。但 AI 有两个主要短板：一是对项目已有数据结构不敏感（如不了解 db.json 中字段的实际命名），容易产生字段名不匹配的问题；二是倾向于"过度工程化"（如建议引入 VeeValidate 等第三方表单校验库）。Day4 阶段必须依靠人工审查来确保 AI 生成代码与现有项目的一致性。

九、遇到的问题与解决

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| `isFormValid` computed 有副作用 | 在 computed 中调用 validate 函数会修改 `formErrors` 状态（Vue 不推荐 computed 有副作用） | 将按钮禁用判断改为调用无副作用的 `checkFormValid()` 函数，将校验逻辑移到 `handleSubmit` 中执行 |
| FormField 组件 class 不生效 | FormField 自身的 `.form-half` class 需要通过 `$attrs.class` 透传 | 在模板中直接使用 `class="form-half"` 绑定到 `<FormField>` 上，Vue 自动合并 class |
| 切换类型后旧数据的 image 未清除 | `resetForm()` 中调用了 `removeImage()`，但 watch 中也调用了 `resetForm()` | 确认流程正确：watch → resetForm() → removeImage()，图片在类型切换时正确清除 |
