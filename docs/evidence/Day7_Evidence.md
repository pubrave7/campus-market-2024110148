## 六、AI 协作复盘

### 6.1 AI 在哪些阶段帮助最大

1. **页面骨架快速生成**（Day2）：AI 一次性生成了 8 个页面骨架组件，节省了大量手动创建和编写基础模板的时间
2. **Mock 数据设计**（Day3）：AI 生成了真实感强的校园场景示例数据（商品名称、价格、描述等），比手工编造更快更自然
3. **表单校验与交互逻辑**（Day4）：AI 能快速编写表单校验规则和数据提交流程
4. **组件化开发**（Day6）：LoadingState、ErrorState、SearchBar 等通用组件的快速实现
5. **文档整理**（Day7）：README、CHECK_REPORT、证据卡等文档的快速整理

### 6.2 AI 生成内容中出现过的问题

1. **超越功能边界**：AI 会生成 JWT Token、权限路由、密码加密、支付功能等超出项目范围的代码——需要人工识别和删除
2. **TypeScript 类型不严格**：使用 `any` 类型逃逸类型检查，需要人工修复
3. **中文编码问题**：AI 生成的 db.json 中文数据出现编码损坏，需要手动修复并添加 UTF-8 声明
4. **文档"编造"风险**：AI 在生成 README 时会描述未实现的功能（如"支持图片上传"、"支持实时聊天"）——必须逐项审核
5. **过度设计**：有时会生成比需求更复杂的实现方案，反而增加理解和维护成本

### 6.3 自己如何判断和修改 AI 生成内容

- **类型安全审查**：运行 `npm run type-check` 和 `npm run lint` 识别问题并修复
- **功能边界判断**：对照实验手册，确认哪些功能在范围内、哪些超出
- **业务逻辑验证**：手动测试每个功能的完整流程，确保数据流转正确
- **代码可读性审查**：检查变量命名、组件划分、注释是否清晰
- **安全性考量**：确保密码不存储在 localStorage、不暴露敏感数据

### 6.4 哪些内容必须由自己理解后才能完成

- **业务需求分析**：理解校园场景下的用户需求，拆解为具体功能
- **技术方案选择**：决定使用 Pinia 还是其他状态管理方案、localStorage 还是其他持久化方式
- **功能边界划定**：判断 Mock 注册登录与生产级认证的区别，避免引入不必要的复杂度
- **质量把控决策**：修复什么、保留什么、什么时候停止添加新功能
- **项目展示表达**：用自己的语言讲清楚项目做了什么、为什么这样做

### 6.5 以后使用 AI Coding 工具时会注意什么

1. **先理解需求，再使用 AI**：不能让 AI 替代理解决策，首先自己理解透彻
2. **每次生成后立即审查**：不等 AI 生成大批代码后再审查，而是边生成边审查
3. **善用 TypeScript 和 Lint**：利用工具自动检测 AI 代码的类型问题和规范问题
4. **保持功能边界意识**：明确告诉 AI 什么不需要（如"不要生成 JWT 认证代码"）
5. **文档必须逐句核实**：AI 生成的文档内容必须与代码实际行为一一对照
6. **保留人工判断痕迹**：在证据卡中记录哪些是 AI 生成的、哪些是自己修改的

## 八、项目不足与改进方向

### 当前不足

| 问题 | 说明 |
|------|------|
| Mock 数据服务 | 使用 JSON Server，非真实后端，数据在项目间不互通 |
| 登录方案 | Mock 登录，无密码加密、无 JWT Token、无服务端校验 |
| 收藏持久化 | 收藏数据存储在 Pinia 内存中，刷新页面后丢失 |
| 图片功能 | 使用 emoji 代替真实图片，无上传功能 |
| 搜索功能 | 仅支持关键词匹配，不支持模糊搜索和多条件组合筛选 |
| 消息中心 | 仅展示静态会话列表，无实时通信功能 |
| 响应式设计 | 基本支持移动端，但部分页面在手机上的体验可进一步优化 |
| 测试覆盖 | 未包含单元测试和 E2E 测试 |

### 改进方向

1. **接入真实后端**：替换 JSON Server 为 Node.js/Spring Boot 等真实后端服务
2. **完善认证体系**：实现 JWT Token 认证，添加密码加密
3. **收藏持久化**：将收藏数据同步到 JSON Server 或 localStorage
4. **图片上传**：接入文件上传服务，支持真实图片展示
5. **搜索增强**：实现模糊搜索、拼音搜索、多条件筛选
6. **消息系统**：实现基于轮询或 WebSocket 的实时消息
7. **移动端适配**：优化小屏幕下的布局和交互
8. **测试覆盖**：添加 Vitest 单元测试和 Playwright E2E 测试

## 九、今日反思

通过 7 天实训，我对前端项目开发、AI 协作开发和工程化过程有了全新的理解。

**关于前端开发：** 以前我可能认为前端开发就是"写页面"，但通过这 7 天，我理解了前端开发是一个系统工程——从业务分析到页面结构设计、从数据建模到 API 封装、从状态管理到交互优化，每一步都需要严谨的思考和决策。Vue 3 的 Composition API、Pinia 的响应式状态管理、TypeScript 的类型系统，这些工具不是为了让代码"看起来高级"，而是实实在在地提高了代码的可维护性和可靠性。

**关于 AI 协作：** 这次实训让我对 AI Coding 工具有了更清醒的认识。AI 确实能大幅提高开发效率——比如生成页面骨架、设计 Mock 数据、编写表单校验逻辑——但它不是万能的。AI 会犯错：会生成超出范围的代码、会产生类型安全问题、会"编造"不存在的功能。真正的工程质量把控必须由人来完成——需求判断、功能边界划定、代码审查、测试验证，这些都依赖开发者的专业判断。**AI 是工具，不是替身。**

**关于工程规范：** Git 提交记录、Daily Evidence、代码规范检查、构建验证——这些看似"额外"的工作，实际上是保证项目可持续性的关键。没有规范的 Git 提交，开发过程无法追溯；没有证据卡，学习过程无法评估；没有 Lint 和类型检查，代码质量无法保证。工程规范不是负担，而是一种对项目负责任的态度。

**最大的收获：** 我学会了如何"管理"一个前端项目——从 0 到 1 搭建、逐步迭代、在 AI 辅助下保持代码质量、最终形成可演示可验收的成果。这种"端到端"的项目经验，比任何单个技术知识都更有价值。
