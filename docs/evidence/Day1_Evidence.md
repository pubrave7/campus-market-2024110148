## 二、项目背景理解（Task 1）

### 1. 为什么要开发校园轻集市？

校园轻集市是一个面向校园场景的轻量级二手交易平台。开发它的目的是：

- **满足校园内二手交易需求：** 学生们有大量闲置物品（教材、电子产品、生活用品等）需要流通
- **降低交易门槛：** 相比校外平台（闲鱼、转转），校园内的交易更可信、更方便
- **实践前端工程化开发：** 通过真实项目学习 Vue 3 + TypeScript 工程化开发流程
- **体验 AI 辅助开发：** 在实践中学习如何有效使用 AI Coding 工具

### 2. 系统主要解决哪些实际问题？

- **信息不对称：** 买家和卖家在校园内找不到彼此，需要一个集中的信息平台
- **交易信任问题：** 校园实名制 + 校内交易，相比校外平台更安全
- **物品利用率低：** 毕业季大量物品被丢弃，通过平台可以流转给需要的同学
- **沟通效率低：** 传统 QQ 群/微信群消息杂乱，缺乏结构化的商品展示

### 3. 系统包含哪些业务场景？

- **商品浏览：** 首页展示商品列表，支持搜索与分类筛选
- **商品发布：** 用户上传商品信息（图片、描述、价格）
- **商品详情：** 查看商品详细信息，决定是否购买
- **用户系统：** 注册、登录、个人信息管理
- **交易沟通：** 买卖双方通过平台沟通交易细节
- **收藏管理：** 收藏感兴趣的商品以便后续查看

### 4. 七天实训最终要完成什么？

通过七天的实训，完成一个功能完整的校园轻集市前端应用，同时：

- 掌握 Vue 3 工程化开发
- 掌握 Git 版本管理
- 积累 AI Coding 协作经验
- 理解软件工程规范
- 完成过程性证据记录
- 通过项目验收

### 思考

**1. 为什么采用这种目录结构？**

这种目录结构遵循 Vue 3 社区的最佳实践（如 Vue 官方脚手架 create-vue），按功能职责划分目录。好处是：
- 职责清晰：api 管接口、components 管组件、stores 管状态、views 管页面
- 易于协作：团队成员可以并行开发不同目录
- 便于维护：新增功能时知道代码应该放在哪里

**2. 每个目录负责什么？**

- `api/` — 封装后端 API 请求，统一管理接口调用
- `components/` — 存放可复用的 UI 组件（如商品卡片、导航栏）
- `router/` — 定义 URL 路径与页面的映射关系
- `stores/` — 使用 Pinia 管理全局状态（如用户信息、购物车）
- `views/` — 页面级组件，一个文件对应一个页面

**3. 哪些目录需要自己开发？**

`src/api/`、`src/components/`、`src/stores/`、`src/views/` 以及 `src/router/` 的扩展都需要自己开发。

**4. 哪些目录属于课程基础设施？**

`docs/`、`scripts/`、`public/` 以及根目录的配置文件都是课程基础设施，提供了统一的开发基线。

## 四、核心代码阅读（Task 3）

### main.ts 分析

```typescript
import { createApp } from 'vue'      // Vue 3 应用创建函数
import { createPinia } from 'pinia'  // Pinia 状态管理创建函数
import App from './App.vue'           // 根组件
import router from './router'         // 路由实例

const app = createApp(App)  // ① 创建 Vue 应用实例
app.use(createPinia())      // ② 注册 Pinia（状态管理）
app.use(router)             // ③ 注册 Router（页面路由）
app.mount('#app')           // ④ 挂载到 DOM
```

**关键理解：**
- Vue 3 采用"创建实例 → 注册插件 → 挂载"的三步启动流程
- `app.use()` 是插件注册的标准方式
- `#app` 对应 `index.html` 中的挂载点
- Pinia 和 Router 都是通过插件机制集成
- 项目入口是 `index.html` → `main.ts` → `App.vue`

### Router 分析

```typescript
import { createRouter, createWebHistory } from 'vue-router'

const router = createRouter({
  history: createWebHistory(import.meta.env.BASE_URL),  // HTML5 History 模式
  routes: [
    { path: '/', name: 'home', component: HomeView },   // 首页路由
  ],
})
```

**Router 的作用：**

Router 是单页应用（SPA）的核心——它让不同 URL 显示不同页面组件，但整个过程不会刷新浏览器。本质是一个"URL ↔ 组件"的映射表。

**为什么首页能显示？**

1. 用户访问 `http://localhost:5173/`
2. 浏览器加载 `index.html` → 执行 `main.ts`
3. Vue 应用启动，Router 检测 URL 为 `/`
4. 匹配到 `path: '/'` 对应的 `HomeView` 组件
5. 在 `App.vue` 的 `<RouterView />` 处渲染 HomeView
6. 页面显示"项目启动成功"

### App.vue 分析

```vue
<script setup lang="ts"></script>
<template>
  <main class="app">
    <h1>校园轻集市</h1>
    <p>AI 辅助前端工程实践种子项目</p>
    <RouterView />  <!-- 🔑 关键：路由出口，匹配的页面组件在此渲染 -->
  </main>
</template>
```

**App.vue 的作用：**

App.vue 是整个应用的"根容器"：
- 提供全局布局框架（标题、导航等）
- `<RouterView />` 是页面内容的渲染出口——路由匹配到的组件会替换这里的占位
- 所有页面都在 App.vue 内部渲染，形成统一的页面结构

## 九、实验思考

### 1. 为什么课程没有要求第一天立即开发页面？

第一天不写代码，而是理解项目、分析目录、阅读核心代码，是因为：

- **磨刀不误砍柴工：** 不理解项目结构就写代码，会导致代码放错位置、破坏既有规范
- **建立全局观：** 先看清"地图"再走路，避免后期返工
- **熟悉基础设施：** 理解哪些是课程提供的，哪些需要自己开发

### 2. 软件工程中的"理解项目"为什么比"开始编码"更重要？

- 编码是"怎么做"，理解项目是"做什么"和"为什么这么做"
- 不理解项目就编码 = 没有地图就出发，容易走错方向
- 软件工程的成本主要在维护阶段，良好的理解是高质量代码的前提
- 实际工作中，阅读和理解代码的时间往往超过编写代码的时间

### 3. AI 在软件开发中应该承担什么角色？

- **辅助者而非替代者：** AI 擅长解释、生成模板、查漏补缺；但不擅长理解业务上下文
- **加速器而非方向盘：** AI 可以加速开发，但技术决策和架构设计需要人来把控
- **学习伙伴：** AI 是很好的"随时可问的老师"，帮助快速理解不熟悉的技术

### 4. 今天最大的收获是什么？

最大的收获是建立了一个清晰的"项目地图"——知道了每个目录是干什么的、应用是怎么启动的、路由是怎么工作的。在这个基础上，后续开发就知道代码该写在哪里，出了问题也知道去哪里排查。这种"全局理解"比学会某个具体 API 更重要。
